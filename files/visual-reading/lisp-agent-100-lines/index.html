100 行 Lisp Agent：
当技能变成记忆

25 年前，教授把 Lisp 称作“AI 的语言”。作者如今重新发现：它未必替模型负责推理，却非常适合表达模型外面的递归、工具生成与能力重放。

8行递归 loop
1个工具：eval
20行左右的记忆
≈100行完整 agent
00 / TL;DR

真正有趣的不是短，而是三层能力被压成同一种材料

循环

agent 的核心只是把消息历史交给模型；如果模型请求工具，就执行、追加结果，再递归一次。

能力

Lisp 的代码与数据使用同一种结构，模型拿到 eval 后，可以在运行时写出新的函数，而不只调用预装工具。

记忆

函数会随进程消失，但写函数的对话会留在 transcript。新进程读回历史，就可能把能力重新求值出来。

边界先说清：这是作者放在本地 Docker 容器中的玩具实验。任意 eval 等于任意代码执行；完整 transcript 也会不断挤占上下文窗口。

01 / RETURN

教授说对的，不是旧 AI 的胜负

作者回忆 2000 年左右的 AI 课程：当时的 Lisp 与符号 AI 绑在一起，后来统计方法和深度学习成为主流，Lisp 也像是随旧范式一起退场。

约 2000
“AI 的语言”

课堂强调符号、规则和递归。作者真正留下的记忆，是 Lisp 作为一种优雅的程序表达。

后来
符号 AI 失势

旧答案没有赢，但 Lisp 的核心特性并没有因此失去意义。

2026
推理由模型承担，外环仍是程序

现代 agent 恰好需要把消息、代码和结果不断送回自身，这与 Lisp 擅长的程序即数据重新对上。

作者重新评价的不是“符号 AI 应该获胜”，而是 Lisp 作为 agent substrate 的形状。

02 / RECUR

先把框架剥掉：Agent 是对消息历史做递归

无论外层产品多复杂，作者把最小 loop 还原成一个二分判断：模型已经回答，或者模型还想调用工具。

输入messagesₙ到目前为止的完整历史
→
模型call-model返回文字或 tool calls
文字回答Base case追加最终消息并返回
请求工具Recursive case执行、追加、再次进入 loop
messagesₙ₊₁↺
原文结构，压缩转写 
(agent-loop messages)
  → model 返回 message
  → 有 tool-calls：执行并递归
  → 无 tool-calls：返回新历史
  1. 状态没有另建容器：messages 参数就是状态。
  2. 停止条件很具体：没有工具请求，递归结束。
  3. 工具结果进入同一条历史：下一轮模型因此看见刚刚发生的动作。
为什么作者更愿意把它看成递归，而不是 while loop？

两种写法都能工作。递归的表达优势在于：当前消息历史作为参数进入函数，工具调用产生更丰富的历史，再把它交给同一个函数；最终回答自然成为 base case。

03 / EVAL

唯一工具不是计算器，而是一门可被模型改写的语言

Lisp 的 homoiconicity 让程序本身也是列表。于是作者没有逐项增加工具，只暴露了一个很薄的入口：读取模型写出的 Lisp form，再对它求值。

( )
核心动作 
(eval (read-from-string form-string))

模型交付的不是某个工具参数，而是一段可以定义函数、组合库、继续改变运行时的程序。

能力观固定工具目录作者的 Lisp 实验
谁决定能力设计者预先编写并发布模型在任务发生时决定并编写
调用单位已批准的工具 contract可以被求值的 Lisp form
能力边界目录明确，治理相对清晰开放且强大，也意味着任意代码风险
作者关注点“我给 agent 装了什么”“agent 发现自己还需要什么”
原文第一个例子：模型现场计算 Fibonacci 30

作者要求模型计算而不是背答案。模型先在 live image 中定义递归函数，再调用它得到 832040。这个小例子为后面“现场写出 web search”做了铺垫。

安全约束

eval 让模型运行任意代码。作者明确只在本地 Docker 容器中运行这个玩具，没有把它当作可直接部署的生产方案。

04 / REMEMBER

记忆先不做检索系统，只把运行时写下来

消息已经是 hash table 的列表，结构上接近 JSON。作者因此直接把 transcript 序列化到文件，下一次启动再读回同一种运行时结构。

01recall从 memory.json 读历史
→
02append加入本轮用户消息
→
03recur进入同一个 agent-loop
→
04remember把新历史写回文件

Recall, recur, remember.读取、递归、记住。

这里没有 schema、migration 或 store abstraction。简洁来自“序列化格式就是运行时格式”，但它也没有遗忘策略。

这 20 行左右的记忆解决了什么，又没有解决什么？

它解决跨进程保留完整对话，例如今天告诉 agent 名字，明天重新启动仍能回答。它没有控制历史规模，transcript 会持续增长，最终碰到上下文窗口。作者设想可以在 recall 与 loop 之间加入压缩步骤，但没有把它实现成本文结论。

05 / GROW

关键转折：它用唯一的工具，给自己造出第二个工具

作者把临时 Brave Search API key 放进对话后，agent 检查现有库、写 HTTP 调用、解析 JSON，并把 brave-search 定义进 live image。

初始能力eval

没有预装 web search。

→
观察环境HTTP + JSON 库

模型发现已有 dexador、shasht 与 URL 编码能力。

→
运行时生成brave-search

新函数被定义进当前 Lisp image。

→
任务结果实时搜索回答

工具目录从 1 项变成 2 项。

传统平台通常在设计时决定 agent 能做什么；这个实验把决定推迟到运行时。能力不再只是一份预装清单，也可能是对话中刚刚写出的程序。

06 / REHYDRATE

进程忘了函数，transcript 记得它是怎么被造出来的

defun 只存在于当前进程，退出后函数消失；但函数源码和创建过程仍在持久化对话里。作者由此看见“技能作为记忆”的机制。

交互图解

模拟一次能力重启

  1. 01能力被写出

    brave-search 已存在于进程 A。

  2. 02进程退出

    live image 消失，函数本体不在磁盘。

  3. 03历史被读回

    memory.json 仍保存定义函数的对话文本。

  4. 04能力重新求值

    新进程沿历史再次执行定义，skill 回到运行时。

transcript+eval→rehydrated capability

“an agent whose skills are literally stories it tells itself.”

作者把这个现象概括为：技能成为 agent 对自己讲述、并能再次执行的故事。
07 / SKILLS

“技能即记忆”是一条能力光谱，不是对 MCP 的简单否定

作者把固定 contract 与运行时生成放在两个端点，并认为行业已经通过 code interpreter、SKILL.md 等形式逐步向中间移动。

设计时决定运行时决定
01MCP / 固定工具

能力作为预先协商、版本化和批准的 contract。

02Code interpreter

为一次问题写临时程序，结果比程序更重要。

03SKILL.md

能力以文本保存，在需要时加载进上下文。

04Transcript skill

agent 在对话中写出能力，并把创造过程本身当作可重放记忆。

保留作者的谨慎语气

把人类理解成“自身实践的 transcript”是作者主动标明的延伸联想，不是实验证据。页面把它保留为思想启发，而不是系统设计定律。

08 / LIMITS

别把实验的诗意，误读成生产方案的完成度

01

任意代码执行

eval 的开放能力与风险是同一件事。作者只在隔离的本地容器里运行。

02

记忆无限增长

完整 transcript 没有遗忘与预算，持续累积会撞上模型上下文窗口。

03

能力治理尚未回答

实验展示运行时生成的可能性，没有替代固定 contract 在批准、版本和可审计性上的价值。

04

这是一个原型观察

最强结论是“Lisp 很适合描述 agent 外环”，不是“所有 agent 都应该改用 Lisp”。

09 / TAKEAWAY

模型负责推理，Lisp 负责让推理结果可以再次成为程序

这篇文章的回环很漂亮：教授当年把 Lisp 与 AI 绑定，预测的技术路线没有赢；但 Lisp 最深的能力，也就是程序可被检查、变换并送回自身，却恰好描述了现代 agent 的工程外环。

100 行只是入口。真正值得带走的问题是：当 agent 可以把经验保存成文本、再把文本执行成能力时，“工具”“记忆”和“技能”的边界还剩多少？
