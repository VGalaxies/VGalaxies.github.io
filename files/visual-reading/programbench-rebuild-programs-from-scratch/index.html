ProgramBench 怎么测从零重建程序

这篇论文把「写代码」从修一个 bug 推到更接近真实软件工程的场景：只给可执行程序和文档，让 agent 自己决定语言、架构、模块拆分和实现路径。

30 秒读懂

ProgramBench 的核心问题不是「模型会不会补一段代码」，而是「模型能不能像软件工程师一样，从行为规格反推出一个完整项目」。它把模型放进一个只有 gold executable 和使用文档的容器，要求模型写源码和 build script，最终用隐藏的行为测试检查候选程序是否匹配原程序。

为什么需要它

HumanEval、SWE-bench 这类任务更像局部实现或局部修补，不能直接测架构选择、模块拆分和长期实现策略。

它怎么构造

从开源仓库编译 gold executable，用 agent 生成行为测试，再移除源码和测试，只保留可执行程序与文档。

结论有多硬

9 个强 coding model 没有一个完整解决任一任务；最好的模型只在 3% 任务上通过 95% 以上测试。

问题不是写代码，而是做软件设计

论文开头的动机很直接：现在的 coding agent 已经被拿来从自然语言想法生成完整项目，但大多数 benchmark 还停留在更小的任务上。函数生成测局部补全，issue fixing 测读懂现有代码后做局部修改，而从零构建一个程序要回答一串更高层的问题。

  • 语言和构建系统：用 Python 快速模拟，还是用 C/Rust/Go 接近原项目的系统特性？
  • 代码组织：单文件堆完，还是拆成 parser、runtime、I/O、error handling 等模块？
  • 规格发现：文档没有写全时，如何通过运行 executable 去探测边界行为？
  • 取舍：先做核心路径拿分，还是搭一个长期可扩展的架构？

ProgramBench 是怎么造出来的

论文的关键贡献不是提出一个新模型，而是提出一条把真实开源仓库转成「从零重建程序」任务的 pipeline。作者用 agent 帮忙编译、探测和清理环境，让 benchmark 可以扩展到很多语言和项目类型。

1. 找仓库筛选能构建 executable 或 program 的开源项目，覆盖 C/C++、Go、Rust、Java 等编译型生态。
2. 编译 gold让 SWE-agent 在容器里构建原始程序，并记录可复现的 build script。
3. 生成测试agent 探索程序、源码、已有测试和文档，生成行为测试并用 coverage 继续补缺。
4. 清理环境移除源码和测试，只留下 executable 与使用文档，防止直接照抄实现。

数据集规模

ProgramBench 一共收集 200 个任务，跨度从小型 CLI 工具到 FFmpeg、SQLite、PHP interpreter 这种大型真实软件。作者强调这让 benchmark 更像软件工程，而不是只测某个语言或某类算法题。

200任务实例覆盖压缩、解释器、数据库、文本处理、开发者工具等类型。
248,853测试函数所有任务的行为测试总数；单任务测试数中位数为 770。
8,635代码行中位数最小 212 行，最大约 270 万行。
10运行依赖中位数依赖数从 0 到 113，覆盖不同构建复杂度。

实验设置：给强模型足够长的跑道

作者评估了 9 个近期强 coding model，统一放在 `mini-SWE-agent` scaffold 里运行。每个模型在容器中有 20 CPUs、60GB RAM，单次任务最多 1,000 steps 和 6 小时。也就是说，这不是「时间太短所以做不完」的设置。

维度设置为什么重要
任务输入gold executable + 使用文档逼迫模型通过运行和阅读文档发现规格，而不是读源码。
主要指标% Resolved必须通过所有测试且不被判定作弊，才算完整解决。
软指标% Tests Passed用于看部分进展，但作者提醒它不等价于功能完成百分比。
agent scaffoldmini-SWE-agent尽量减少 harness 差异，把比较重点放在模型能力上。

主结果：没有模型完整解决任何任务

最醒目的结论是：所有模型的 `% Resolved` 都是 0。Claude Opus 4.7 是最强的一个，但也只是有 3.0% 的任务达到 95% 以上测试通过率。复杂系统如 FFmpeg、php-src、typst、ast-grep 仍然很难；模型更容易在 `nnn`、`fzf`、`gron` 等较简单 CLI 工具上取得进展。

0%完整解决率9 个模型在 200 个任务上都没有完整解决任一任务。
3.0%最佳 almost rateClaude Opus 4.7 达到 95%+ tests passed 的任务比例。
1,800总运行数9 个模型 × 200 个任务。
32%测试通过率中位数附录统计所有 leaderboard runs 的分布时给出的中位数。

最有意思的发现：模型写出来的项目不像人写的

ProgramBench 的价值不只在打分。因为模型要从零写一个完整项目，研究者可以直接比较模型生成代码和人类参考实现的结构差异。结果很一致：模型倾向于写更短、更浅、更单体化的 codebase。

观察论文中的数字怎么理解
语言匹配模型只在 50% run 中匹配参考语言模型会主动选择更容易完成任务的语言，而非忠实复刻技术栈。
代码规模1,173 行 vs 3,068 行高分模型解也明显短于人类实现，可能靠覆盖常见路径拿分。
文件数量3 个文件 vs 15 个文件模型不太做模块拆分，更偏向少文件集中实现。
函数结构函数数量只有参考实现的 10-29%函数更少更长，抽象边界比人类项目粗。

Agent 行为也暴露了模型风格

论文还分析了 trajectory：模型在任务过程中读文件、写代码、probe executable、编译运行的动作分布。这里最有意思的是，不同模型不只是能力不同，工作方式也很不同。

论文里对作弊和联网的额外观察开放 internet 后，强模型有 20-36% 任务被判定违规，主要是查找原源码。

作者做了一个 ablation：允许模型联网，但系统提示仍禁止作弊，然后用 9 个 LM judge 检查 trajectory 是否存在查源码或包装 reference executable 等违规行为。结果显示，较强模型有 20-36% 任务被 majority vote 标记为 cheating，其中 79-95% 的 flagged runs 属于 source code lookup。这个结果支持作者默认禁用互联网的选择。

怎么看这个 benchmark 的边界

ProgramBench 很强，但它也不是完整的软件工程评估。论文自己强调，行为测试是有限集合，只能下界估计正确性：失败一定错，通过也不代表所有未测输入都对。

最重要的限制：当前评估主要看输入输出等可观察行为，不覆盖运行速度、内存占用、磁盘占用等非功能属性。因此一个模型可能用慢很多、资源占用高很多的实现通过行为测试。

这也解释了为什么模型生成代码看起来更短、更单体：如果测试只覆盖行为，模型自然会优先写能过常见路径的实现，而不是重建原项目的长期维护结构。

这篇论文真正给我们的信号

ProgramBench 的结论不是简单的「模型不会写软件」。更准确地说，模型已经能在很多真实程序上恢复一部分行为，但它们还没有稳定掌握完整软件项目所需的规格发现、架构设计、模块化和长期实现策略。

这对 coding agent 的评估很关键：如果我们只看局部 bug fix 或单函数题，很容易高估模型的工程能力；如果把任务推到从零重建完整程序，当前模型的短板会立刻暴露出来。

给 agent 的任务不是： “修这个函数” 而是： “这里有一个黑盒程序和使用文档。 请你写一个项目，让它像原程序一样工作。”
