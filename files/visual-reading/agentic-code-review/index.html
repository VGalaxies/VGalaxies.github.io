代码写得越快，为什么审查越重要？

Agent 把“写出来”的成本压低了，却没有让“看懂、证明、负责”的成本同步下降。于是软件工程最稀缺的能力，从生成代码迁移到了建立可信度。

WRITE FAST ? TRUST SLOW

真正的交付单位不是“生成了多少代码”，而是“有多少代码被证明可以工作”。

30 秒 TL;DR

AI 没有消灭工程瓶颈，只把它向下游推到了验证。最有效的做法不是让人类逐行看完所有代码，也不是让 Agent 闭环自我批准，而是按错误代价分层：确定性检查守底线，异构 AI reviewer 做规模化分诊，人类拥有目标、高风险关口与最终合并责任。

01 / BOTTLENECK SHIFT

写代码变便宜了，理解没有

过去审查之所以能自然跟上，是因为读代码比写代码快。Agent 打破了这个速度关系：机器可以瞬间制造大 diff，但人的阅读、推理与信心建立速度几乎没变。

02 / RISK SPECTRUM

“应该审查多少”没有统一答案

审查强度取决于三个变量：失败会伤到谁、代码要活多久、需要多少人共同理解它。用同一套规则管理周末原型和支付系统，只会在一端制造摩擦，在另一端制造事故。

建议的人类审查强度 轻量 保留测试与最终确认

没有用户，允许延后部分审查

把注意力放在真实测试、关键路径和不可逆操作上。代码可能很快被重写，重复和局部粗糙的长期成本较低，但“少审查”仍不等于“不验证”。

危险的是中间地带。项目开始有用户、团队开始扩张，但流程仍沿用“我一个人知道就够了”的习惯。此时 review 的两个作用会同时开启：它既要抓 bug，也要把系统知识分发给其他维护者。

这不是无解的人类速度问题，而是可以修复的工具问题。PR 在进入 review 前，至少应携带：

  • 目的：为什么要做这个改动，它解决什么用户或系统问题。
  • 决策：考虑过哪些方案，为什么选择当前实现。
  • 证据：实际运行了什么测试，输出是什么，哪些假设还没有被证明。
  • 边界：改动刻意没有覆盖什么，最可能在哪里失败。
04 / HETEROGENEOUS REVIEW

不要寻找唯一最好的 reviewer，要寻找互补的盲区

AI reviewer 已经能发现大量人类会漏掉的问题，但不同工具并不会稳定地盯住同一类缺陷。高风险场景的价值来自异构性，而不是把同一种模型复制四份。

从线上影响看问题，优先暴露会造成事故或大面积退化的路径。

利用仓库知识识别重复实现、约定冲突和跨文件依赖。

05 / REVIEW PIPELINE

让廉价检查先发生，把人类留给昂贵判断

可扩展的 review 不是给所有 diff 叠满检查，而是按风险逐层加码。越靠前的门越便宜、越确定，越靠后的判断越依赖上下文、责任与产品目标。

从机器确定性到人类责任

风险越高，向右走得越远

GATE 01

准入证据

目的、范围、测试输出、决策记录。没有证据的 PR 不进入昂贵 review。

GATE 02

确定性 CI

类型、lint、测试、覆盖率、安全规则。它们不会被自信的解释说服。

GATE 03

AI 分诊

按改动范围、文件类型和历史模式预测风险，快速放行小而明确的变更。

GATE 04

异构审查

用不同偏好的 reviewer 检查正确性、架构、安全与生产故障风险。

GATE 05

人类拥有合并

确认目标正确、未写下的需求、高爆炸半径路径，以及最终责任。

变更类型确定性门AI reviewer人类投入
文档、格式、机械配置lint、构建、链接检查一遍轻量审查快速确认范围
普通产品逻辑类型、单测、集成测试一主一辅，关注正确性看意图、测试改动与边界
认证、支付、数据迁移完整 CI、安全与回滚验证两个异构 reviewer，加安全检查系统 owner 深审并拥有合并
06 / PRACTICAL PLAYBOOK

把文章压缩成六条可执行规则

这些规则都在保护同一种稀缺资源：能够理解系统并对结果负责的人类注意力。

01按风险分层，不按作者分层同样是 Agent 写的代码，文档改动和支付路径不能接受同样的审查深度。›

先问“错了会怎样”，再决定需要多少门。低风险变更快速通过，高风险变更叠加测试、异构 reviewer、系统 owner 和安全审查。

02提前熔断高维护 PR用 patch 大小、文件类型和范围等便宜信号，先识别会吞噬审查时间的变更。›

33,707 个 Agent PR 的研究显示，小而明确的任务更容易快速合并。把大而模糊的变更退回拆分，不要让 reviewer 先投入一小时才发现 Agent 无法完成反馈循环。

03没有证据，就不开始 review要求目的、范围、测试输出和实际运行证明，把意图恢复成本推回提交者。›

准入门槛不是官僚流程，它阻止 reviewer 成为“第一个看见代码的人”。PR 描述应让陌生维护者在读 diff 前先知道这次改动想证明什么。

04刻意保持 PR 小可读性现在是架构约束，小 diff 才能让人类真的参与，而不是被迫盖章。›

让 Agent 按独立意图拆提交，限制每次改动的文件和目标。大 diff 容易被直接拒绝，也容易在疲劳下得到没有理解的批准。

05先读测试改动，再读实现Agent 可能通过改写断言来让错误行为变绿，覆盖率也不能证明测试能抓住错误。›

大量测试被修改本身就是风险信号。检查断言是否仍表达原需求，必要时使用 mutation testing 验证测试是否真的会在实现出错时失败。

06把 AI review 当传感器，不当裁决模型可以发现信号，但目标是否正确、未写下的需求和最终责任仍属于人。›

平静而自信的“looks good”只是一个读数。最终点击 merge 的人要能解释改动为什么存在、如何被证明，以及出问题时谁来处理。

07 / HUMAN ON THE LOOP

人类没有离开，只是上移了一层

逐行阅读所有 diff 的时代正在结束，但这不等于让 Agent 写、Agent 审、Agent 判定后自动闭环。相关模型可能共享盲区，多个“通过”也可能只是相关性很高的借来信心。

Human in the loop

人类逐个参与生成与审查，在高吞吐量下很快成为队列瓶颈。

→

Human on the loop

人类设计门、抽样、审计、处理升级，专注目标与高代价错误，并始终拥有最终合并。

不能转交给模型的四件事：

  • 责任：模型不会在凌晨三点被叫醒，也不能为事故负责。
  • 目标判断：代码正确，不代表这就是应该构建的改动。
  • 高爆炸半径关口：错误代价昂贵时，需要真正理解系统的人做决定。
  • 未被写下的行为：模型擅长审查存在的代码，不擅长发现从未有人写进需求的空白。
TAKEAWAY

团队的优势，不再是生成最多代码

真正长期有效的优势，是建立一套能把“看起来正确”转换成“被证明可以工作”的审查系统，并且永远不把测试变绿误认为有人理解。
