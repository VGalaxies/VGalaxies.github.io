Loop 不是让 Agent 一直跑

真正的 loop，是把任务设计成工作、验证、修正、停止的闭环。你交出去的不是一句更长的 prompt，而是一部分控制权。

四类 loop 是自治阶梯

从交出检查、停止条件、触发时机，到交出稳定的重复工作流。

越自治，边界越要明确

成功标准、轮数、频率、模型、脚本和 usage 观测缺一不可。

01 · LOOP ANATOMY

先别问用哪个命令，先回答四个问题

Claude Code 团队给出的定义很朴素：Agent 重复工作周期，直到停止条件成立。差异不在“有没有循环”，而在循环的四个控制点由谁掌握。

TRIGGER 谁启动下一轮？

人发一条 prompt、计时器到点，还是外部事件出现。触发方式决定 loop 是否需要人在场。

WORK 一轮做哪些事？

读上下文、修改代码、运行工具、回复消息。工作单元越清楚，偏航越容易发现。

VERIFY 怎样证明变好了？

测试数量、性能分数、页面交互、队列是否清空。可量化检查比“看起来不错”更可靠。

STOP 什么时候必须停？

目标达成、达到最大轮数、工作已合并，或等待更多上下文。没有停止条件，就没有可控 loop。

验证不通过 把失败信息带进下一轮，继续修正
下一轮？
目标达成或边界触发 退出，并交付结果、状态和剩余风险
02 · AUTONOMY LADDER

四种 loop，是四次控制权交接

这不是四个互相排斥的产品功能，而是一条从人工协作到主动运行的阶梯。点击查看每一级你交出了什么，又必须保留什么护栏。

人在场，Agent 做一轮完整工作

Turn-based：先交出“检查”

每条 prompt 都会启动一次 agentic loop。Claude 收集上下文、行动、检查并回复，你再决定下一步。

  • 适合：短任务、探索、需要人持续判断方向的工作。
  • 停止：Claude 判断完成，或发现缺少上下文。
  • 增强：把人工验收流程写成 SKILL.md，让 Agent 能自行看见、操作和测量结果。
检查页面交互
运行测试与性能审计
控制台必须无新增错误
失败后修复，并从头复验
交给 Agent执行和端到端检查
仍由人控制何时发起下一轮
仍由人控制方向和“够不够好”的最终判断
成本边界写具体 prompt，减少来回轮数
人在场，Agent 迭代到可验证目标

Goal-based：再交出“何时算完成”

当一次执行不够时，明确成功条件，让 evaluator 在每次 Agent 想停止时重新检查。未达标就继续，达标或触及轮数上限才退出。

  • 适合：有确定退出条件的复杂任务。
  • 停止：目标达成，或达到最大尝试次数。
  • 最有效的标准：测试全部通过、错误数量归零、分数达到阈值。

最多尝试 5 轮
交给 Agent多轮执行和修正
交给 evaluator按条件判断是否达标
仍由人控制目标定义和最大轮数
成本边界阈值越明确，越少浪费在主观判断
不必盯着，按时间检查外部世界

Time-based：继续交出“何时开始”

有些工作内容不变，只是输入持续变化。有些任务要等 PR review、CI 或队列状态。最简单的接口，就是按时间间隔检查变化并响应。

  • 适合：周期摘要、PR 跟进、CI 修复、外部队列处理。
  • /loop：在本机按间隔重跑，机器关闭就停止。
  • /schedule：把 routine 放到云端，按计划执行。

处理新 review，并修复失败的 CI
交给时钟到点自动触发下一轮
交给 Agent读取外部状态并采取行动
仍由人控制间隔、取消条件和权限范围
成本边界频率匹配真实变化速度，不做无效轮询
无人值守，稳定流程持续运转

Proactive：最后交出“整条重复工作流”

当输入流稳定、处理步骤清楚、成功标准可验证时，可以组合 schedule、goal、skills、dynamic workflows 和 auto mode，形成长期运行的主动例程。

  • 适合：bug 报告、issue triage、迁移、依赖升级等持续工作流。
  • 停止：单个任务完成即退出，routine 持续到被关闭。
  • 模型路由：快而便宜的模型做常规处理，强模型做判断和审查。

交给 routine触发、分派、执行和反馈
交给 workflow并行探索与多 Agent 协作
仍由人控制权限、预算、升级路径和总开关
成本边界先小规模 pilot，再扩大输入和 Agent 数
03 · CHOOSE THE SMALLEST LOOP

不是选最强的 loop，而是选最小的闭环

原文反复强调：不是所有任务都需要复杂 loop。下面的选择器用任务特征给出最小起点，目的是降低自治和成本，而不是追求功能最多。

04 · QUALITY SYSTEM

输出质量不只来自模型，更来自 loop 外面的系统

同一个模型放进不同仓库、验证方式和 review 流程，结果会完全不同。原文把质量责任放回系统设计：代码库提供惯例，skill 定义验收，独立 reviewer 提供新鲜视角。

代码库惯例 干净的结构、清楚的命名和一致的模式，是 Agent 最先模仿的东西。
可执行验证 让 Agent 能看见、操作、测量结果，而不是只凭编辑成功就宣布完成。
新鲜上下文审查 第二个 Agent 不受主 Agent 推理路径影响，更容易发现盲点。
把失败写回系统 一次问题变成下一次的规则、测试、脚本或 skill，loop 才会持续变好。
01 验证要尽量量化“按钮能点”比“页面看起来不错”更容易自证。 +
把测试通过数、性能阈值、控制台错误数、截图前后差异等写进 skill。检查越可观测，Agent 越能闭环。
02 文档必须触手可及外部最佳实践会变化，别让 Agent 只靠旧记忆。 +
为框架、库和内部规范提供可访问的最新文档入口，让 loop 在每轮决策时都能核对真实约束。
03 审查者要有独立视角不要让主 Agent 自己给自己的推理背书。 +
用独立上下文的 reviewer 检查代码、风险和完成标准。对高风险任务，让 reviewer 能拒绝或要求重做。
04 修系统，不只修这一次重复出现的问题说明 loop 缺少护栏。 +
把失败转成测试、脚本、文档或 skill。否则下一轮只是用更多 token 重新犯同一个错误。

写清达到什么条件停止，以及最多尝试多少轮，防止过早或过晚退出。

06 · START SMALL

从一件“你正在成为瓶颈”的工作开始

别先设计宏大的自治系统。找一件你反复介入的具体工作，识别你究竟卡在检查、停止条件、触发时机，还是整条流程。

找到瓶颈

回看最近一周：哪件事总在等你检查、催促、重新发 prompt，或重复执行同一套步骤？

我反复在做什么？

只交出一个控制点

先把验证写成 skill，或把成功阈值写进 goal。不要第一天就交出触发、权限和多 Agent 编排。

我能先交出哪一块？

观察卡住与越界

运行后记录它在哪里停滞、过度行动或浪费 token，再把观察结果写回停止条件和验证系统。

下一轮怎样更可控？
TAKEAWAY

设计 loop，本质上是在设计责任边界

先让 Agent 看得见“什么叫完成”，再让它多跑几轮。先证明一个小闭环可靠，再把更多触发、判断和流程交出去。

这篇文章最实用的提醒是：自动化的起点不是更复杂的编排，而是你能否明确说出自己正在交出哪一部分工作。
