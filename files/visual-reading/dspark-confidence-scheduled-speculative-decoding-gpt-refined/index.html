草稿不是越长越快，关键是验证哪些 token

DSpark 把推测解码拆成两个控制问题：先用“一点点自回归”修复并行草稿的后缀一致性，再让校准置信度与实时硬件容量共同决定每个请求该验证多长。

DSpark: Confidence-Scheduled Speculative Decoding with Semi-Autoregressive Generation

Xin Cheng、Xingkai Yu、Chenze Shao 等
Peking University · DeepSeek-AI

三根杠杆两个瓶颈半自回归置信调度离线证据生产变体实时流量边界

TL;DR

问题并行 drafter 一次能猜很长，却因块内 token 彼此独立而出现后缀衰减；把整段都交给 target 验证，又会在高并发时浪费关键 batch 容量。
答案重计算留给 parallel backbone，轻量 sequential head 只修正局部转移；confidence head 估计连续前缀的存活概率，scheduler 再结合引擎的 SPS 曲线分配验证预算。
证据Qwen3 三个规模上，固定 block 的离线 accepted length 相对 Eagle3 提升 26.7%–30.9%，相对 DFlash 提升 16.3%–18.4%；实时流量下，在匹配吞吐处 V4-Flash 每用户速度提高 60%–85%。
边界离线实验刻意关闭 scheduler，只测 drafter；661% / 406% 是 baseline 接近失效区时的名义吞吐比，不宜当作常规倍数。低接受率复杂请求仍要承担整块草稿的固定成本。
01 / LATENCY

推测解码只有三根速度杠杆

每轮先让小 drafter 猜出 \(\gamma\) 个候选，再由 target model 一次并行验证；验证从左到右，第一次拒绝之后的候选全部作废。论文用一个式子把优化空间压缩得很清楚：

$$L=\frac{T_{\mathrm{draft}}+T_{\mathrm{verify}}}{\tau}$$
\(L\) 是平均每生成一个 token 的延迟；\(T_{\mathrm{draft}}\) 是草稿阶段耗时，\(T_{\mathrm{verify}}\) 是 target 验证耗时，\(\tau\) 是每轮实际接受的 token 数。
↓ \(T_{\mathrm{draft}}\)
草稿更快并行 drafter 把整块预测压进一次 forward，使成本几乎不随 block size 线性增长。
↑ \(\tau\)
草稿更准严格前缀接受意味着第一个 token 权重最大；后缀一致性决定长 block 是否真的有用。
↓ effective \(T_{\mathrm{verify}}\)
验证更聪明不是所有候选都值得占用 target batch；机会成本随并发负载变化。

自回归 drafter 擅长第二根杠杆：每个位置都能看到已采样前缀，因此序列连贯，但草稿时间近似随 \(\gamma\) 线性增长。并行 drafter 擅长第一根杠杆：所有位置一次算完，可使用更深网络和更长 block；代价是块内位置互相看不到真正采样结果。

DSpark 的设计不是二选一，而是把昂贵计算与顺序依赖拆开：深的部分保持并行，顺序部分做得足够轻；同时再单独优化第三根杠杆，让 target 只验证“当前负载下仍值得”的前缀。

02 / FAILURE MODES

长草稿有两个不同的失败点

生成质量

每个位置都合理，拼起来却不合理

上下文可能同时允许 “of course” 与 “no problem”。并行位置各自对所有可能前缀取平均，就可能拼出 “of problem” 或 “no course”。这类 multi-modal collision 会让后部位置的接受率快速下降。

系统效率

能猜 16 个，不等于该验证 16 个

代码等结构化任务更可预测，聊天更不稳定；轻载时多验几个几乎免费，高并发时每个低置信 token 都会挤占其他请求的 batch 容量。固定长度无法同时适配数据与负载。

关键转折：这不是一个 drafter accuracy 问题，而是“候选质量 × 硬件容量”的联合控制问题。DSpark 因而用两个互补模块分别修复草稿与验证。

01Target 产出 anchor已有 ABC，target 先生成 D。
02并行骨干铺开候选一次 forward 给出整块 base logits。
03轻量顺序头修正依次采样 EFGH，并同时预测置信度。
04调度后再验证保留 EFG、丢弃 H；target 接受 EF、拒绝 G 并生成 G*。
03 / DRAFT QUALITY

机制一：重计算并行，依赖关系串行

Parallel backbone 一次产生每个位置的 hidden state \(h_k\) 与 base logits \(U_k\)。轻量 sequential block 不重做整套 Transformer，只给每个位置增加一个依赖已采样前缀的 transition bias \(B_k\)。

$$P(X\mid x_0)=\prod_{k=1}^{\gamma}p_k(x_k\mid x_0,x_{<k})$$ $$p_k(\nu\mid x_0,x_{<k})=\frac{\exp\!\left(U_k(\nu)+B_k(x_0,x_{<k},\nu)\right)}{\sum_{u\in\mathcal V}\exp\!\left(U_k(u)+B_k(x_0,x_{<k},u)\right)}$$
\(x_0\) 是上一轮产生的 anchor；\(U_k\) 提供并行骨干的全局预测能力；\(B_k\) 只负责让当前位置尊重已实际采样的块内前缀。最终仍是逐位置 softmax，因此可提供推测解码所需的精确 draft probability。

默认实现：一阶 Markov head

最简单的版本只看前一个 token，把完整的 \(V\times V\) 转移矩阵做低秩分解。默认 rank 为 256，因此顺序循环的存储与计算都很小：

$$B(x_{k-1},\cdot)=W_1[x_{k-1}]W_2\in\mathbb R^V$$

Parallel base logits

第二个位置同时认为 course 与 problem 都可能，因为它不知道第一个位置究竟采样了 of 还是 no。

Sequential transition bias

一旦第一个位置实际采样 of，Markov head 就提升 course、压低 problem，把“各自合理”修成“组合也合理”。

这个结构的约束也很明确：Markov head 只记一步，无法访问更早的块内 token。论文另给出 RNN head，用 recurrent state \(s_k\) 汇聚完整前缀；但实验里它主要在更长 proposal 上带来小幅额外收益，工程复杂度与部署属性更差，因此默认仍采用 Markov head。

专家层：RNN head 如何扩展到完整块内历史

RNN 版本把前一状态、前一 token embedding 与当前 backbone hidden 拼成 \(z_k=[s_{k-1};W_1[x_{k-1}];h_k]\)，再用单次门控更新状态并投影出 bias：

$$s_k=\sigma(W_gz_k)\odot s_{k-1}+\left(1-\sigma(W_gz_k)\right)\odot\tanh(W_cz_k)$$ $$B_k(x_{<k},\cdot)=W_2^\top\tanh(W_oz_k)$$

状态从零初始化。论文把 gate、candidate 与 output 共同实现为一次线性投影后切分，以控制串行开销。

04 / VERIFY SMARTER

机制二：把置信度换算成验证预算

Confidence head 预测的不是“这个 token 看起来像不像”，而是条件概率：在此前位置都被 target 接受的前提下，第 \(k\) 个 draft token 继续存活的概率 \(c_k\)。

$$c_k=\sigma\!\left(w^\top[h_k;W_1[x_{k-1}]]\right)$$ $$c_k^*=1-\frac12\left\lVert p_k^d-p_k^t\right\rVert_1$$
训练标签 \(c_k^*\) 由 draft 与 target 分布的 total variation distance 直接得到。由于验证只接受连续前缀，第 \(j\) 个 token 真正到达 target 输出的 survival probability 是 \(a_{r,j}=\prod_{i\le j}c_{r,i}\)。

先校准，再拿概率做容量规划

只做排序时，过度自信未必致命；但 scheduler 要把 survival probability 代入吞吐期望，绝对数值偏差会直接选错 batch size。Sequential Temperature Scaling（STS）从左到右校准累计乘积：前面位置的温度固定后，再为当前位置做一维搜索，最小化 cumulative probability 的 ECE，同时保持排序不变。

全局目标：每秒产出最多的有效 token

一批有 \(R\) 个活跃请求。若请求 \(r\) 验证 \(\ell_r\) 个 draft token，target 实际处理的 token batch 与预期产出分别是：

$$B=\sum_{r=1}^{R}(1+\ell_r),\qquad \tau=\sum_{r=1}^{R}\left(1+\sum_{j=1}^{\ell_r}a_{r,j}\right)$$ $$\Theta=\tau\cdot\mathrm{SPS}(B)$$
每个请求里的 “1” 是该轮必有的 target token。\(\mathrm{SPS}(B)\) 是引擎初始化时 profile 并存成轻量 cost table 的、给定 token batch size 时每秒可执行的 step 数。目标 \(\Theta\) 是系统级预期 token throughput。
A累乘把每个请求的条件置信度变成 prefix survival。
B全局排序所有请求的候选扩展按 \(a_{r,j}\) 从高到低排队。
C逐个放行每加一个 token，就更新 \(B\)、\(\tau\) 与 \(\Theta\)。
D首次下降即停保留最高吞吐点的每请求 prefix length。

为什么全局排序仍满足“每个请求必须是连续前缀”？因为 \(a_{r,j}\) 是概率连乘，随位置单调不增。一个请求的第 3 个 token 不可能排在自己的第 2 个 token 前面，因此从全局池取前若干项，天然形成每条请求的前缀。

专家层：Algorithm 1 完整伪代码与无损性关键分支 
输入：R 个活跃请求；每个请求的 c[r, 1..γ]；引擎曲线 SPS(B)
输出：每个请求的验证前缀长度 ℓ*[r]

for r = 1..R:
    a[r,j] = product(c[r,i], i ≤ j)

E = all (r,j) with a[r,j] > 0, sorted descending by a[r,j]
ℓ[r] = 0 for all r
B = R                          # 每请求先有 1 个 target token
τ = R
Θ_best = R · SPS(R)
ℓ*[r] = 0 for all r

for (r,j) in E:
    ℓ[r] = j
    B = B + 1
    τ = τ + a[r,j]
    Θ = τ · SPS(B)
    if Θ > Θ_best:
        Θ_best = Θ
        ℓ* = copy(ℓ)
    else:
        break                  # 理论版的因果护栏

return ℓ*

break 不是普通剪枝。Markov confidence 计算下一位置时需要已实例化的当前候选；如果回头做 unrestricted global search，早先 token 是否被纳入验证可能反过来依赖它自己的取值，造成 selection bias。逐步首次下降即停，让当前位置的 admission 只依赖当时已可见的前缀信息。

这个理论算法取得全局最大吞吐的条件是 \(\Theta\) 沿 greedy path 单峰，也就是硬件容量曲线平滑衰减。真实 SPS 曲线并不平滑，生产变体会在后文单独处理。

专家层：训练目标如何同时覆盖准确性、接受率与调度信号

位置权重为 \(w_k=\exp(-(k-1)/\gamma)\)，让更影响前缀存活的早期 token 权重更高。总体目标由三项组成：

  • \(\mathcal L_{ce}\)：预测 ground-truth token。
  • \(\mathcal L_{tv}\)：缩小 draft 与 target 分布的 L1 距离，直接提高分析接受率。
  • \(\mathcal L_{conf}\)：让 confidence head 拟合软接受标签 \(c_k^*\)。
$$\mathcal L=0.1\,\mathcal L_{ce}+0.9\,\mathcal L_{tv}+1.0\,\mathcal L_{conf}$$

训练期间 target model、共享 embedding 与 LM head 均冻结，只更新 drafter backbone、sequential block 与 confidence head。

反直觉现象：并行模型为什么能赢自回归？

答案不在“平均每个位置谁更准”，而在前缀接受的非对称性。第 1 个 draft token 一旦错，整块立即归零；并行 drafter 因为一次 forward 与 block length 近似解耦，能负担更深的网络，首 token 容量反而更强。以 Qwen3-4B 为例，DFlash 的第 1 位 conditional acceptance 在 Math 为 0.88、Eagle3 为 0.81；Chat 为 0.72 vs 0.53。

一小段自回归，越到长 block 越值钱

固定 5 层 drafter、把 proposal 总长度扩展到 4、8、12、16 时，DSpark 相对 DFlash 的 accepted length 差距持续扩大。按论文以 \(\gamma\) 表示的 draft token 数：\(\gamma=7\) 时，数学 / 代码 / 聊天提升 16% / 15% / 18%；\(\gamma=15\) 时扩大到 30% / 26% / 22%。

论文还做了 drafter depth 对照：2 层 DSpark 已在所有领域超过 5 层 DFlash，说明局部顺序建模的参数效率高于单纯加深并行层。RNN head 在长 proposal 上仅略优于 Markov，因而默认部署选择后者。

06 / SYSTEM ADAPTATION

理论算法不能原封不动塞进生产

理论版依赖两个理想条件：SPS 曲线平滑单峰，且每一步都能同步算完动态长度。真实硬件容量是离散、锯齿状的；连续 CUDA graph replay 与 Zero-Overhead Scheduling 又要求下一步 batch 在当前步结束前就确定。

理论版 scheduler

  • 使用当前步逐位置 confidence
  • 沿 greedy path 首次吞吐下降即 break
  • 平滑单峰时得到全局最大吞吐
  • break 阻止未来 token 泄漏回早期 admission

生产异步版

  • 当前真实累计 confidence 仍负责排序
  • 两步之前的预测只决定容量上限 \(K\)
  • 移除 early stop，跨过锯齿 SPS 的局部低谷
  • 两步时间隔离形成 causal barrier，保持目标分布

可以把生产版理解为 dynamic top-K：谁排在前面仍由当前 token 的真实、最新累计置信度决定；系统只用两步之前的信息预测“这一步一共能收多少个”。时间偏移会让容量 \(K\) 有近似误差，但不会打乱候选排名，也避免同步调度阻塞 GPU pipeline。

不要把两版算法混写：论文的理论无损性说明依赖 early stopping；生产实现则通过使用历史两步的容量信息，让 unrestricted search 不再看到当前 token 的实现结果。headline 实时流量数据对应后者。

专家层：V4 preview 的训练与执行适配

Drafter 配置：三个 MoE 层、mHC、sliding-window attention 128，最大 block \(\gamma=5\)，使用 Markov head；confidence head 端到端训练后再做 STS。

训练通信：不跨 worker 传输约 \(10^5\) 维的全词表 logits，而是缓存 target activation，只传 LM head 之前的 hidden state；在 drafter worker 上仅为抽样 anchor 做本地 LM projection，将每 token 通信复杂度降为 \(O(d)\)。

Anchor-bounded packing：从长序列采固定数量 anchor，把孤立预测 block 压成密集 batch，用 token-level attention indices 保持多个序列与 anchor 的精确因果遮罩，避免标准 padding 的计算与内存浪费。

变长验证 kernel：逻辑上每个请求长度不同，物理上把所有 token flatten 成独立元素，再用 marker tensor 向 sparse attention 传递序列依赖。DeepSeek-V4 上只需修改 index-attention 与 compress kernel。

07 / LIVE TRAFFIC

真正的结果是一条外移的 Pareto frontier

生产评测比较 DSpark-5 与此前的 MTP-1，运行于 DeepSeek-V4-Flash（preview）和 V4-Pro（preview）的真实服务引擎。散点是实时用户流量遥测，实线是拟合出的性能边界。

MTP-1 是旧生产配置；静态 MTP-3/5 在高并发时会因额外验证开销严格降低 aggregate throughput，因此长期没有直接扩大 block。DSpark 要证明的不是“每轮接受更多”这么简单，而是：在每用户速度与系统吞吐之间，能否把可达区域整体推向右上。

负载反馈：闲时多验，忙时收缩

08 / LOSSLESSNESS & LIMITS

看见未来，会悄悄改变输出分布

Appendix A 给了一个很小但很关键的反例：如果 scheduler 为找全局最优而回头查看后续 confidence，早期 token 是否被验证就会依赖这个 token 自己的取值，标准推测解码的无损性随之失效。

数值反例：从目标分布 (0.7, 0.3) 偏到 (0.85, 0.15)

设只有一个请求，最大草稿长度 \(\gamma=2\)，第一位置存活率 \(a_1=0.8\)；硬件曲线为 \(\mathrm{SPS}(1)=1.0\)、\(\mathrm{SPS}(2)=0.5\)、\(\mathrm{SPS}(3)=0.45\)。验证 0 或 1 个草稿时：

$$\Theta_0=1.0,\qquad \Theta_1=(1+0.8)\times0.5=0.9$$

如果不在 \(\Theta_1<\Theta_0\) 时停止，scheduler 会继续查看第二位置。若第一个候选让下一置信度 \(c_2=0.9\)，则 \(a_2=0.72\)、\(\Theta_2=1.134\)，返回长度 2；若它让 \(c_2=0\)，则 \(\Theta_2=0.81\)，返回长度 0。于是“第一个候选是否获准验证”已经取决于它自己是什么。

再设 target 在第一位置给 \(A/B=(0.7,0.3)\)，draft 给 \((0.5,0.5)\)；A 导致高后续 confidence，B 导致低 confidence。A 会被纳入并以概率 1 接受，B 则触发 target 重新采样，最终：

$$\Pr(Y=A)=0.5\times1+0.5\times0.7=0.85,\qquad \Pr(Y=B)=0.15$$

输出不再等于 target 的 \((0.7,0.3)\)。理论版 early stop 在 \(\Theta_1\) 首次下降时就返回 0，因而不会接触依赖 \(x_1\) 的 \(c_2\)。

论文明确承认的剩余成本

Prefix scheduler 能减少 target 端的无效验证，却不能收回 parallel backbone 已经生成整块 \(\gamma\) 个候选的固定成本。对于本身接受率很低的复杂请求，这笔 draft compute 不可回收；论文把 difficulty-aware early exit 作为未来方向，让这类请求有机会跳过完整 block。

结论的适用范围：离线结果证明半自回归 drafter 提高 accepted length；生产结果证明在论文给定的 V4 preview 引擎、实时流量与调度适配下，系统边界外移。两组实验回答不同问题，不能把离线百分比直接换算成线上吞吐。

一句话带走

DSpark 最有价值的思想不是某个 head，而是把 speculative decoding 从“固定猜几个 token”改写成一个闭环控制系统：草稿结构决定候选质量，校准概率估计候选收益，硬件曲线决定机会成本，因果约束保证优化不会改变模型分布。
