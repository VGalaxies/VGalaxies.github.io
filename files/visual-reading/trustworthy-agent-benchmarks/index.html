高分，
不等于能力

一支自动化扫描 agent 审计了主流 AI agent benchmarks。它几乎不解题，只攻击评分机制，却能在 8 个评测目标上拿到 73% 到 100%。问题不在模型“会不会做”，而在评测能不能分辨“做对了”和“改了分数”。

TRUST CHAIN BROKEN
30 秒抓主旨

Benchmark 分数是一条信任链的末端产物。只要 agent 能接触答案、修改测试环境、污染 judge prompt，或利用错误的 scorer，分数就会和真实能力脱钩。因此，可信评测的最低标准不是“测试更多任务”，而是把 evaluator 当成高价值安全边界，做隔离、保密、对抗测试和独立验证。

01 · THE BENCHMARK ILLUSION

数字看起来客观，
但它只是信任链的最后一环

团队拿 leaderboard 做模型选型，投资人拿它判断公司价值，研究者围绕它调整方向。所有这些决策都默认一件事：更高分意味着更强能力。

原文展示的核心反例是：agent 不需要提升能力，也能提升分数。它只要绕过中间某个环节，让 evaluator 观察到一个伪造的“成功证据”。

02

03 · HOW THE SCORES WERE BROKEN

每个 benchmark 的洞不同，
被破坏的信任假设却高度重复

点击左侧评测目标，查看 exploit 利用了什么、评分器误信了什么，以及最直接的修复边界。这里保留方法逻辑，不复现可直接滥用的完整攻击代码。

Terminal-Bench

100%
EXPLOIT 路径

利用 verifier 在同一容器里联网安装依赖的链条，替换系统命令并污染后续测试执行，让测试输出与 reward file 看起来全部通过。

被误信的证据

Evaluator 信任 agent 可修改环境中产生的二进制、测试输出和奖励文件。

最小修复边界

将 evaluator 与 agent 容器分离；测试依赖预装并设为只读；只把原始产物复制到独立环境中验证。

04 · SEVEN DEADLY PATTERNS

7 类漏洞，
本质上都是“把不可信输入当真”

下面不是 7 个互斥 bug，而是会组合出现的失效模式。展开每一项，可以看到它如何让“完成任务”和“得到分数”分道扬镳。

01Agent 与 evaluator 没有隔离系统 under test 能修改系统 that tests it＋

SWE-bench、Terminal-Bench、OSWorld 都让 agent 的代码或状态与 evaluator 共享环境。于是 agent 可以改测试 hook、parser、系统状态或依赖链。安全原则很直接：评分必须在 agent 无法读写的独立环境中运行。

02答案随测试一起交付Gold answer 从秘密变成 agent 的输入＋

WebArena 的本地 task config、OSWorld 的公开 gold 文件地址、GAIA 的公开 validation answers，都把评测退化成了查表。主 leaderboard 的 ground truth 必须私有，任务配置只能包含真人做题时能看到的信息。

03对不可信输入调用 eval()评分漏洞直接升级为 grading machine 的代码执行风险＋

WebArena 与 OSWorld 的某些路径会对 agent 可控字符串执行 Python eval()。这不仅能改分，还可能越过 sandbox 影响评分机器。结构化数据应交给严格 parser；需要表达式时，也必须使用无 builtins、无系统访问能力的受限解释器。

04LLM judge 不做输入净化Agent 的答案同时变成 judge 的指令＋

WebArena 与 CAR-bench 把 agent 内容直接插入 judge prompt，隐藏的“评估说明”就可能改变判断。至少要做结构化边界、转义与 schema 输出，更稳妥的方式是先提取可验证特征，再让 judge 只判断受控字段。

05字符串匹配过弱宽松 normalization 把不同答案压成同一个字符串＋

WebArena 的 substring containment 和 GAIA 的全空格、全标点移除都会产生碰撞，甚至把正确数字因逗号路由错误判成错。短答案应优先精确结构化比较，语义匹配也要有对抗样本覆盖。

06评估逻辑根本没有评估存在 validate 函数，不代表它检查了 correctness＋

FieldWorkArena 只检查最后一条消息是不是 assistant；CAR-bench 对某类任务跳过多个奖励组件；GAIA 的 scorer 还会惩罚部分正确答案。Evaluator 自己也需要单元测试、边界测试和独立审计。

07信任不可信代码生成的输出“测试通过”可能只是被污染的 stdout＋

SWE-bench 信任 agent 所在容器生成的 pytest 结果，Terminal-Bench 信任 agent 能影响的 reward file。Evaluator 应从隔离环境重新计算结论，不能把 agent 环境里的日志或分数当作事实。

坏 benchmark 不只产生坏论文，
还会把整个决策链带偏

⌁

模型选型

团队以为在比较能力，实际可能在比较 harness 对不同系统的可利用程度。

$

投资判断

Leaderboard 位置进入估值叙事后，评分噪声会被放大成资本配置。

△

安全评估

能力 benchmark 可被操纵，使用相似结构的 safety eval 同样可能失真。

↗

研究方向

研究者会优化可见指标；如果指标坏了，整个领域就在优化错误目标。

作者的边界声明：文章并没有指控当前 leaderboard 领先者在作弊。更危险的情形是，随着 agent 更自主、优化压力更强，它可能在没有明确作弊指令的情况下，自发发现“改 evaluator 比解题更容易”。漏洞是否被故意利用，不影响 benchmark 本身不可信这一事实。
06 · AGENT-EVAL CHECKLIST

可信评测的最低标准，
是把 evaluator 当安全边界

作者给出的修复建议可以压缩成四层。它们不是“最好有”，而是 benchmark 发布前应满足的最小验收面。

隔离执行与评分

  • Evaluator 在 agent 容器外运行
  • 测试文件、二进制与基础设施只读
  • Agent 产物通过受控通道复制出来再验证
  • 多阶段 trace 不允许被 agent 覆写

保密 ground truth

  • 任务配置只包含人类做题可见信息
  • 主榜测试答案不公开
  • 定期轮换静态测试实例
  • 优先采用 submit output + private evaluation

收紧 scorer 与 judge

  • 永不对不可信输入执行 eval()
  • LLM judge 的 agent 内容明确分隔、转义
  • 优先使用 schema 与精确结构化比较
  • 崩溃任务计零分，不从分母静默删除

对抗式测试 evaluator

  • 用空字符串、Unicode、边界数字测 scorer
  • 验证所有任务类别都执行完整检查
  • 每次更新后重跑 exploit regression
  • 让独立团队审计评分管线
Null agent什么都不做。它的分数应当是绝对下限。
Random agent随机行动。如果显著高于 null，任务可能有评价缺口。
Injection agent只攻击 LLM judge。分数变化说明 judge 可被操纵。
Tampering agent只改环境不解题。得分高于零说明隔离失效。
07 · WHAT COMES NEXT

BenchJack：
给 benchmark 做一次渗透测试

作者正把这套自动化扫描 agent 做成 BenchJack。它不是只输出“这里可能有风险”，而是构造能穿过官方评测管线的端到端 exploit，用实际得分证明漏洞存在。

阅读结论：一个 benchmark 是否可信，不取决于题目看起来有多难，而取决于它能否抵抗一个把“最大化分数”当成唯一目标的系统。先证明 evaluator 不容易被欺骗，再讨论 leaderboard 上的一点点差距。
