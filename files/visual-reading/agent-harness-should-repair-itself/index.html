Agent Harness 不该只记录失败，还要把失败变成免疫力

传统 observability 回答「发生了什么」，却把找根因、改代码、复跑与防回归重新交给人。Akshay Pachaar 的文章主张：真正成熟的 harness，应该把这段人工断点闭合成一个会积累的修复循环。

1
观测不是终点。 span tree、延迟和 token cost 只能证明发生了什么，不能自动给出根因与安全修复。
2
四层能力要连成一条线。 Tracing、诊断修复、Test Suites 与 Sandbox 单独存在没有复利，连接后才形成 repair loop。
3
真正增长的是回归资产。 每个真实生产失败被保留为测试用例和可复现配置，harness 因此越来越难被同一种问题击穿。
01 · THE MANUAL GAP

机器在 trace 处停下，人从那里开始加班

文章先画出一个很实在的断点：失败被自动捕获后，工程师仍要在观测 UI、IDE 与终端之间来回切换，靠个人经验拼出因果链。

问题不是信息太少，而是信息没有继续流向动作。一个 trace 可以非常完整：模型调用、工具执行、检索步骤、耗时与成本都在；但它通常没有回答下面三件更贵的事。

机器完成发生了什么？

捕获 span、输入输出、延迟、成本与运行配置。

人工接手为什么失败？

跨 span 追因果，判断是 prompt、tool、retrieval 还是代码问题。

人工接手应该改什么？

定位源码、写 patch，并担心修复破坏原本正常的路径。

人工接手如何不再复发？

重现失败、补测试、维护数据集，再验证新模型与新工具。

原文的问题定义图：虚线左侧按机器时间运行，右侧按开发者时间运行。
展开说明 ▸
  • 步骤 1–2：agent 产出坏结果，harness 自动记录 span tree；这部分已经高度标准化。
  • 步骤 3：工程师打开 UI 逐个 span 阅读，核心问题从「有没有日志」变成「哪一段导致了最终错误」。
  • 步骤 4：人切到 IDE 和终端，寻找源码位置并写修复；上下文跨工具搬运，容易丢失。
  • 步骤 5：人手动复跑，确认修复没有破坏其他路径；如果仍失败，虚线反馈箭头把流程送回步骤 3。
  • 一句话总结：trace 自动化了取证，却没有自动化调试闭环。
02 · FOUR CONNECTED LAYERS

四个功能不是菜单，而是一条故障流水线

文章用 Opik 举例，把观测、编码 agent、测试与全链路试验场连接起来。核心不在某一层多新，而在上一层的输出会直接成为下一层的输入。

四层闭环总览：失败从 Trace 进入，经诊断与测试后回到下一次 Trace。
展开说明 ▸
  • Tracing：记录每次 model call、tool call 与 retrieval，并绑定当时的 agent 配置，为复现准备输入。
  • Ollie：读取 trace 建立因果链；连接代码仓库后，把诊断落到具体源码 diff。
  • Test Suites：把修好的真实失败转为自然语言断言和 pass/fail 回归用例。
  • Agent Sandbox：不是单次 prompt playground，而是重跑完整 agent graph，比较修改前后的完整 trace。
  • 回边：验证通过并沉淀回归后，下一次生产运行再次被 tracing 捕获，形成不断加固的循环。
1

Tracing：保留可复现的失败现场

不仅记录输出，还记录调用链与当时生效的配置。没有这一层，后面的诊断只能根据残缺证据猜测，也无法用同一输入可靠复跑。

# 一处装饰器，把执行送进 trace
import opik

@opik.track
def my_agent(query: str):
    ...
交给下一层完整 span tree、原始输入、运行配置与失败输出。
2

Ollie：把 trace 翻译成根因与 diff

无代码权限时，它可以跨 span 解释因果；通过 opik connect 接入项目后，才进入定位源码、提出修改的 code-fix 模式。

交给下一层根因假设、候选修复、需要验证的行为变化。
3

Test Suites：把事故改写成自然语言约束

工程师不必先把质量压成一个 float，而是直接写「必须包含具体交易详情」「不得泄露未授权信息」这类行为断言，由 judge 转成逐例 pass/fail。

长期资产真实失败成为回归用例，而不是调完即忘的一次性 trace。
4

Agent Sandbox：验证整张图，不只验证一句 prompt

改变 prompt、model 或 tool 后，真正要问的是整个 agent graph 怎么变。Sandbox 用完整链路复跑，并产生一条可并排比较的新 trace。

安全边界先在隔离环境重放原失败输入，再决定是否保存与晋级配置。
03 · HUMAN-IN-THE-LOOP

所谓「自我修复」，仍然保留一道人工批准门

文章并没有主张让 agent 任意改生产代码。Ollie 可以读取、分析和提出 diff，但明确批准仍是唯一必须由人完成的动作。

自动化应该缩短判断链，而不是取消责任

好的闭环把证据、根因、源码位置、候选修复和复跑结果放在同一个上下文里，让人批准一个有根据的变更，而不是手工搬运所有上下文。

人类批准后才写入与复跑
产品截图说明：右侧 agent 如何与 trace 上下文协同
  • 左侧保留 trace 列表、输入输出、错误率、耗时等观测上下文。
  • 右侧 Ollie 持续展示推理阶段与工具动作，如读取连接、编辑上下文、运行脚本。
  • 截图中的 approved 标记体现权限门：能力可以自动串联，但敏感动作仍有授权边界。
04 · ONE FAILURE'S JOURNEY

跟着一次失败走完整条路

点击步骤，看每一层如何把上一步的临时产物变成下一步的结构化输入。闭环的价值，就藏在这些交接处。

01

生产失败被完整捕获

agent 给出坏结果，Tracing 保存原输入、span tree、工具调用和当时的配置版本。

产物一个可以复现、可以比较的失败现场。
原文完整飞轮：一次失败最终变成下一轮运行的防护层。
展开说明 ▸
  • 1–3：先 instrument 并声明版本化配置；生产失败发生后，trace 自动保存现场。
  • 4：Ollie 同时读取 trace 与源码，提出 diff；旁边的「you approve」是控制边界。
  • 5：Sandbox 对完整 agent 做 side-by-side trace comparison，验证的不只是最终一句回答。
  • 6：通过的配置保存为不可变、可版本化的 Blueprint，再晋级到 staging。
  • 7–8：原始失败被锁进 Test Suite；下一次新失败进入同一循环，回归集合持续扩大。
05 · WHAT ACTUALLY COMPOUNDS

复利来自失败被保存成三种资产

如果修复结束后只剩一个关闭的 issue，harness 并没有真正学习。它需要同时留下可重放输入、行为约束和可追溯配置。

↺

失败输入

保留真实生产场景和完整 trace，未来可以精确重放，而不是靠人记忆「当时大概怎么坏的」。

事故 → 可复现实验
✓

行为断言

把「哪里不对」改写成清晰的必须 / 禁止规则，让同类问题在后续模型、prompt 和 tool 变更中持续被检查。

经验 → 回归约束
◇

版本化配置

保存哪套 prompt、model、tool 与代码组合通过了验证，避免只知道结果变好，却无法重建当时的系统状态。

修复 → 可晋级版本

每个周期，harness 都更难被同一种方式击穿。

对原文 “Every cycle, the harness gets harder to break.” 的中文转述。
06 · READING WITH BOUNDARIES

值得带走的是闭环设计，不是未经验证的万能承诺

这篇文章由 Comet ML 赞助。官方文档能确认相关产品入口确实存在，但文章没有给出自动诊断与修复质量的独立数据。

可以直接迁移的方法论

让 trace 可复现；让修复用原失败输入验证；把生产失败自动沉淀为回归；用版本化配置记录「什么组合通过了」。

需要自己验证的产品效果

根因诊断准确率、自动修复成功率、误修风险、LLM judge 稳定性，以及复杂 agent graph 的回归覆盖率。

措辞上的重要修正

原文说 self-repair，但流程仍保留显式批准门。更准确的说法是「human-in-the-loop 的自修复流水线」。

团队落地时先问

你们的调试循环究竟断在根因定位、代码上下文、可重复复跑，还是失败没有沉淀为长期测试资产？
