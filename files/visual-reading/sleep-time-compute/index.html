让 Agent 在睡觉时学习

Agent 干活时会产生大量「废气」——它检索了什么、试了什么、哪里错了、人怎么纠正的。这些是学习的金矿，但形态又乱又多，而且恰好在用户等答案时产生。Context 的做法：别在热路径上学，等空闲算力时再学，就像人睡觉时把白天的经验固化成记忆。

30 秒读懂

Agent 每完成一个任务都会留下一条原始轨迹（raw trail）。里面藏着最值钱的东西——经验教训，但它以最糟的形态、在最糟的时机出现：又脏又长，而且正好是用户在等答案的时候。所以 Context 不在此刻学习，而是把「把轨迹提炼成可复用知识」这件贵活挪到空闲算力上做，避开用户的时钟。这就是 sleep-time compute。

问题

提炼原始轨迹很贵。放在推理时（热路径）做，每个任务都要为上一个任务「打扫卫生」，用户直接感受为延迟。

做法

热路径上只廉价捕获原始轨迹就走人；提炼放到空闲算力上跑，通过 Generator→Reflector→Curator 循环写回上下文层。

结果

明天跑你任务的 agent 比今天更聪明，而今天没有任何一个任务因此变慢——学习「借」的不是工作的延迟预算。

为什么热路径不是学习的地方

推理时有两样东西稀缺：延迟和注意力。用户在等，你花在整理上一个任务记录上的每一毫秒，都是当前任务变慢的一毫秒。当着用户的面把一条乱糟糟的轨迹提炼成干净、可复用的上下文，是笔糟糕的交易。

所以平台在热路径上几乎什么都不做：任务进行时廉价地捕获原始轨迹，然后就走人。真正贵的那部分——把轨迹变成下一个 agent 能用的东西——完全挪到热路径之外，那里没有用户的时钟在跑。

其实只有三条路，另外两条都更糟：

方案 做法 代价
内联提炼 在任务过程中顺手把上一条轨迹整理干净 每个任务都交一笔「打扫」税，用户感受为延迟
干脆不学 捕获了轨迹但从不消化 系统永远不进步，扔掉了每个任务产出的最有价值的东西
热路径外消化 捕获归捕获，提炼挪到空闲算力上做 拿到学习，又不用交延迟税 ✓

为什么「睡眠」这个词用得准

这个类比不只是可爱。生物的记忆固化做的是结构上一模一样的事：白天的原始经验在身体离线时被重放、压缩成持久记忆——恰恰是因为在清醒时做这件事会干扰行动。把固化挪到空闲时间，是同一个动作、出于同一个理由：系统白天行动、夜里学习，两者互不干扰。工作保持快，是因为学习等到了安静的时刻。

趁没人用的时候处理

两个活动任务之间，有算力闲着、没有用户在等。这就是 sleep-time compute 运行的时候。它拿最近工作的原始活动，消化成结构化知识——流程笔记、纠正、反复出现的模式——写回上下文层，供未来的 agent 取用。

那些在热路径上会花掉延迟的工作，在空闲周期上花的是零成本，而且它改善之后的每一次检索。明天的任务更快也更好，靠的是在没人使用的时间里跑完的处理。今天没有任何东西因此变慢。

一个具体例子：财报 deck

说得具体点。一个 agent 在做财报 deck 时留下一条原始轨迹：它打开的文件、画的图表，以及一位专家的纠正——数字应该用调整后 EBITDA、瀑布图应该按季度环比来做。原始形态下，这是几百行事件日志，对下一个 agent 毫无用处。

看这条轨迹如何从一千行日志被压成一句话 展开对比：原始事件日志 → sleep-time compute 写回的一句持久笔记
# 原始轨迹（几百行事件日志，下一个 agent 用不了） [t0] open q3_financials.xlsx [t1] read sheet "P&L", rows 12–48 [t2] draw bar_chart(revenue_by_month) [t3] draw waterfall(net_income, period=YoY) [t4] expert_correction: "use adjusted EBITDA, not net income" [t5] expert_correction: "waterfall should be quarter-over-quarter" [t6] redraw waterfall(adj_ebitda, period=QoQ) ... (还有几百行) ...
sleep-time compute 读完，往这个账户的上下文里写下一条持久笔记：
「本账户使用调整后 EBITDA；瀑布图按季度环比。」
下一个 agent 读的是一句话，而不是一千行日志。

这条管线做了什么

具体来说，消化是一条管线。原始事件被提炼成候选观察；从噪声里抽取信号，保留反复出现的纠正、丢掉一次性的；被保留的信号整理写入上下文存储，成为下一个 agent 在恰当时机读到的结构化、可检索的知识。

SLEEP-TIME COMPUTE 管线
输入 原始工作事件 traces、corrections、observations——又脏又长
Distill 提炼 原始事件 → 候选笔记
Extract 抽取信号 保留反复出现的，丢掉一次性的
Curate 整理入库 写进 .context：结构化、可检索

整条管线跑在空闲算力上，没有用户在等。它的价值在于把工作的原始废气变成结构化上下文，全程避开热路径。

为什么要分成 org / team / individual 三层

管线在三个层级上并发运行，因为「整个组织该学的」和「某个人的助手该学的」是不同的消化结果。属于所有人的模式写在树的高处；只属于某个人的偏好留在他那里。

Org 组织级属于所有人的模式——公司通用规范、跨团队约定，写在树的最高处
Team 团队级某个团队共享的做法与偏好
Individual 个人级只属于一个人的偏好，跟着他走，不污染更高层

核心：Generator、Reflector、Curator

整件事的心脏是一个借自近期研究的循环。它的要点是克制——天真地把 agent 注意到的一切都追加进去，只会让上下文塞满噪声和矛盾，而噪声多的上下文检索得更差，不是更好。这个循环让上下文在信号上增长，而不是在体量上增长。目标不是一个更大的存储，而是一个更锐利的存储。

GENERATOR → REFLECTOR → CURATOR 循环
提议者
Generator

从新活动里提议对上下文的更新——「我觉得这条值得记下来」。

＋ 提出候选
批判者
Reflector

批判这些提议：这条教训是真的吗？和已知的一致吗？值得留吗？

✕ 挡掉噪声
策展者
Curator

决定实际写什么，增量地、一点点地写进存储。

✓ 写入存储
↺ 回流 Curator 写入后，更新过的存储成为下一轮 Generator 的已知背景——所以这是一个循环，不是一次性流水线。存储越锐利，下一轮的提议就越贴合、越少重复。

一致性：Reflector 真正的活

Reflector 真正的工作是保证一致性。当一条新教训和存储里已有的一条矛盾时，把两条都追加进去，会让下一个 agent 拿到两个答案却无从选择。Reflector 抓住这个冲突，Curator 来解决它——要么更新旧笔记，要么把新笔记的适用范围缩小到它真正成立的情况，让存储保持连贯，而不是不断累积矛盾。

写入按语义去重，所以同一条教训学了十次只记录一次——被强化的是它的置信度，而不是它的体量。 —— 目标不是更大的存储，是更锐利的存储

什么时候触发

两个触发器会启动它，一个管稳态，一个管尖峰。关键是：两者都跑在空闲池上，与服务用户的工作负载解耦，谁都不和实时推理抢资源——学习永远不从工作的延迟预算里借钱。

每隔几小时 定时节奏（Schedule） 按固定节拍扫一遍最近的活动，负责稳态消化。
超过阈值 容量阈值（Threshold） 未处理活动堆积到可配置上限时触发，让一波突发工作被及时消化，不用等下一次定时。
0 对实时推理的竞争 两者都跑在空闲池上，与用户工作负载解耦，不占延迟预算。

底层设计：先积累，后呈现

所有这一切底下有一个值得单独拎出来的设计选择，因为它是「own the data, rent the methods（拥有数据、租用方法）」原则在上下文层上的应用。积累知识和呈现知识之间有一条硬边界。

  • 存储 = 持久的积累：组织学到的东西的原始与提炼记录，长期沉淀下来。
  • 呈现 = 可替换的一层：存储如何被编译成某个 prompt 的合适上下文，是独立、可插拔的一层——今天用这套 prompt 优化方法，明天换一套，都不用重新处理底下的存储。

把两者分开的意义就在这里：把上下文变成 prompt 的方法会不断进步，但积累下来的上下文不必每次方法进步时都重建一遍。

回报很具体：当一套更好的 prompt 优化方法上线，你把编译器指向已有的存储，立刻就能拿到更好的 prompt——不用重新处理一年的提炼上下文。你租用最新的方法，保留它所依赖的数据。这就是「上下文层」整个论点，浓缩在一个操作细节里。

在睡梦中完成学习

便宜的洞见是：agent 只要在工作，就会产生用来改进自己的数据。贵的部分是把这些原始数据变成可用的东西——而窍门在于永远不在热路径上为它付费。

  • 廉价捕获：用户在等的时候，只做便宜的捕获。
  • 空闲消化：没人等的时候，在空闲算力上消化。
  • 积累持久、呈现可换：数据长期留存，方法随时替换。
明天跑你任务的 agent 比今天的更好，而你今天没有任何一个任务因此变慢。 —— 这就是「让学习在睡梦中完成」的全部意义
