Working With AI 具体案例 · 解析器回归

会用 AI 修 Bug，为什么还得懂架构？

一次看似普通的解析器回归，连续暴露了 AI 修复的三种常见偏差：只解眼前样例、凭空增加状态、把正确机制放错作用域。

Carson Gross 2026-06-29 htmx Essays 原文链接
30 秒结论

AI 是速度放大器，不是架构所有者。它能快速定位原因、生成覆盖充分的测试；但只有理解既有系统的人，才能判断一个补丁是在解决问题，还是把复杂度悄悄转移到未来。

01 / BUG 现场

一个 as，同时属于两套语法

hyperscript 是一门刻意拥抱英语歧义的 Web 脚本语言。升级到 0.9.91 后，下面这条原本可用的命令突然解析失败：

fetch `{% url 'trade:get_symbol_data' %}?symbol=${symbol}` as JSON

问题不在 JSON，而在解析器决定这个 as 应该跟谁绑定。

预期：表达式只产出 URL，fetch 保留 as JSON 作为自己的响应修饰符。

02 / 根因

AI 很快找到了回归是怎么来的

作者把 go 与 fetch 的 URL 解析抽成共用方法 parseURLOrExpression()。复用本身没错，问题是它无意中扩大了 fetch 后面的语法范围。

01 为了复用而重构

go 与 fetch 共享 URL / 表达式解析。

02 语法边界被放宽

fetch 后不再只读 URL，而是允许一般表达式。

03 as 被提前消费

表达式层把它认成类型转换，而不是 fetch 修饰符。

04 绑定对象改变

字符串先被转 JSON，请求语义因此失真。

这一段正是 AI 的强项：跨提交追踪改动、搜索共享调用点、把回归还原成可解释的因果链。作者说，Claude 几分钟就完成了原本会耗时更久的调查。

03 / 三次提案，一次收窄

代码能跑，不等于修复落在正确的位置

四个方案的差别，不只是“是否通过当前测试”，而是它们分别把特殊性放到了哪一层。逐步切换，观察每次方案留下的隐藏成本。

能修当前样例，但不够一般

方案一：先尝试解析“像字符串的叶子”

Claude 建议先读 stringLike，失败后再回退到一般表达式。它能保住字面量 URL 后面的 as，却遗漏了变量 URL。

return this.parseElement("stringLike")
  || this.requireElement("expression");

// 仍然要支持：
fetch $url as JSON
覆盖范围

只覆盖报告中的字符串式 URL；变量、组合表达式仍有缺口。

人的判断

这是针对样例形状打补丁，而不是恢复语法契约。

更一般，但制造了新状态

方案二：给解析器增加 noConversions

在 URL 解析期间暂时关闭类型转换，AsExpression.parse() 发现标志后直接退出。它解决得更完整，却为解析器增加了一条新的上下文状态。

// AsExpression.parse()
if (parser.noConversions) return;
覆盖范围

能够覆盖一般表达式，但每个读写该标志的地方都成为新的耦合点。

人的判断

项目已经有表达“上层保留关键字”的机制，无需再创造平行基础设施。

机制正确，作用域错误

方案三：利用现成的 follows

作者提醒 Claude：解析器已经能让上层语法“认领”某个后继 token。于是共享方法在解析表达式前后 push / pop as，避免它被表达式层消费。

parser.pushFollow("as");
try {
  return parser.parseURLOrExpression();
} finally {
  parser.popFollow();
}
覆盖范围

既复用现有机制，也支持变量表达式；看起来已经非常接近。

隐藏回归

共享方法也被 go 使用；但只有 fetch 把 as 当修饰符。

现有机制 × 最小语义作用域

最终修复：把特殊性留在 FetchCommand#parse()

as 的冲突只属于 fetch 语法，因此 token 的保留也只应发生在这里。这样既不增加解析器状态，也不破坏 go 中合法的类型转换表达式。

parser.pushFollow("as");
try {
  var url = parser.parseURLOrExpression();
} finally {
  parser.popFollow();
}

if (parser.matchToken("as")) {
  // parse fetch response type
}
为什么更干净

复用既有机制，把例外放在真正拥有该语义的命令层。

关键动作

不是多写一段代码，而是把正确代码从共享层下移到责任边界。

04 / 分工边界

AI 提供吞吐量，人负责守住复杂度

这次协作不是“人想、AI 写”这么简单。更准确的分工是：AI 加速可搜索、可验证的劳动；人判断什么才算符合系统长期利益的答案。

AI / 擅长放大

调查与机械耐力

  • 追踪回归来源从共享方法和历史重构中快速定位根因。
  • 生成小而聚焦的测试覆盖字面量、变量与回归边界，补足人不愿反复 grind 的工作。
  • 快速展开候选方案把可能路径变成可审阅的代码，而不是停留在讨论。
HUMAN / 必须拥有

语义、作用域与技术债判断

  • 知道系统已有哪套机制因此能识别 noConversions 是重复基础设施。
  • 看见共享抽象的副作用因此能发现方案三会误伤 go 的合法语法。
  • 定义“完成”的标准不只要求当前测试绿，还要求责任边界正确、未来复杂度可控。

05 / 巫师，而不是学徒

接受 AI 补丁前，先问五个问题

原文没有把经验写成清单，但这个案例隐含了一套很实用的审查顺序。折叠态先给结论，展开可看它在本例中的具体含义。

01 它修的是语义，还是只修了样例形状？只对字符串有效、对变量失效，就是明显警报。 ＋
把 bug 报告中的输入换一种等价表达：字面量换变量、单一路径换组合表达式。如果补丁立刻失效，它大概率只是 pattern matching。
02 代码库是否已经有表达这个意图的机制？follows 已经存在，就不必再造 noConversions。 ＋
AI 容易基于局部上下文发明一个新开关。熟悉代码库的人应先搜索相邻模块、历史提交与同类语法，优先复用既有概念。
03 这个特殊规则真正属于哪一层？as 冲突属于 fetch，不属于共享 URL 解析器。 ＋
把例外放得太高，会污染所有调用方；放得太低，又可能重复实现。责任边界比“代码最少”更重要。
04 共享调用方会不会被一起改变？修 fetch 时，必须同时验证 go 仍允许合法的转换表达式。 ＋
对共享方法的任何改动，都要枚举所有调用方。方案三之所以“差一点”，就是机制没错，却改变了不该改变的命令。
05 测试是在证明补丁，还是在证明行为契约？同时覆盖原 bug、变量 URL 与 go 的非回归。 ＋
高质量测试不只复刻 bug 报告，还应保护相邻合法行为。AI 很适合生成这些小而聚焦的用例，但用例集合仍需要人定义边界。
06 / 老开发者的旁注

AI 同时补偿衰退，也可能加速衰退

50 岁的 Carson Gross 把 AI 的价值说得很个人：记忆不如从前、连续工作时间变短，而 AI 恰好能帮助恢复上下文并承担枯燥耐力活。但这份便利也让他担心，依赖会让整体智力退化得更快。

它补上的能力

  • 上下文恢复：忘记细节后，仍能借助提示迅速重新理解。
  • 机械耐力：持续生成更完整的测试，不受精力下降限制。
  • 项目切换：在开源与工作项目间更快重建心智模型。
⇄

它可能拿走的能力

  • 主动回忆：每次都外包检索，自己的记忆路径会不会变弱？
  • 独立推演：是否在看完多个提案后，才想起自己本可直接完成？
  • 认知主权：最终判断来自理解，还是来自“模型听起来很确定”？
结论

真正稀缺的不是代码，而是拒绝错误复杂度的判断力

作者没有否定 AI，反而明确肯定它在调查和测试上的价值。他反对的是一种工作方式：把“模型给出的第一个或第二个可运行答案”误当成工程完成。

不是盲目接受方案的“巫师学徒”，而是理解问题、要求答案符合现有架构，并用测试验证它的“巫师”。

把 AI 留在回路里，也把理解留在自己手里。前者提高速度，后者决定速度最终把项目带向哪里。
