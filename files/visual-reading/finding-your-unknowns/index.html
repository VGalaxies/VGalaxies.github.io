Agent 做得越多，越要先找到自己的未知项

Prompt、skills 与 context 只是工作的地图。代码库、真实世界和实施中冒出的边界条件，才是领地。两者之间的差距，就是 Agent 被迫替你猜的地方。

你的 unknowns
TL;DR · 30 秒抓住主旨

能力越强、执行越久的 Agent，会穿过越多你没有说明的决策点。工作质量因此越来越受限于你能否发现自己的未知项。解决办法不是把计划一次写到完美，而是用 blind spot pass、原型、访谈、参考实现、实施笔记、解释文档和测验，在项目全周期持续校准地图。

01 · THE GAP

真正的瓶颈，可能不是模型能力，而是地图里缺了什么

当 Agent 遇到 prompt 没写、skill 没覆盖、context 没提供的决策点，它仍然必须继续。此时它只能根据一般经验，猜一个「你大概会想要」的答案。

地图 目标、prompt、skills、spec、上下文，以及你认为工作会怎样发生。
≠
领地 真实代码、历史包袱、边界条件、用户反应，以及实施后才看得见的约束。

unknowns 不是单纯的「缺知识」，而是地图没有覆盖、但执行又必须做决定的全部空间。

这也是为什么提前计划仍然不够。未知项可能埋在实现深处，也可能反过来告诉你：真正应该解决的根本不是当前这个问题。

作者的核心经验是，和 Fable 协作已经变成一个迭代过程：在实施前、实施中、实施后不断发现 unknowns。换句话说，agentic coding 的重要技能不是「命令写得多细」，而是更早发现哪些地方还不该被默认。

02 · FOUR KINDS

先给未知项命名，才知道该用什么办法找它

四象限的价值不在分类本身，而在于提醒你：有些问题能直接问，有些偏好只能通过看见方案才浮现，还有些盲点连问题都尚未形成。

DIRECT CONTEXT

直接写进地图

这是你已经能表达的部分。把目标、不可变约束、验收条件和当前状态说清楚，Agent 就不必在这些地方猜。

合适手段
  • prompt 与 spec
  • 明确的验收条件
  • 可指向的文件、模块和现有约定
QUESTIONS

让问题变成决策

你知道这里还有空白，所以最适合把它显式化：让 Agent 研究、访谈，或把会改变架构的决策放到实施计划最前面。

合适手段
  • 一次一个问题的 interview
  • 针对性技术调查
  • 强调可变决策的 implementation plan
REACTION

先看见，再把偏好说出来

这类知识很难靠空谈提取。让 Agent 给出差异足够大的方案、低成本原型或真实参考，再从你的反应中提炼标准。

合适手段
  • 多方向 brainstorm
  • 不接后端的 HTML prototype
  • 源码级 reference
BLIND SPOTS

先扩展问题空间

此时继续要求 Agent 「做得更好」通常没有方向。先让它解释领域、搜索代码库历史、列出质量标准与常见坑，帮助你获得提问能力。

合适手段
  • blind spot pass
  • 教学式 explainer
  • 探索历史实现和失败模式
03 · THE BALANCE

太具体会锁死错误路径，太模糊会放大默认假设

指导 Agent 不是在「细」与「粗」之间找一个固定刻度，而是让它理解你的起点，并知道哪些地方应该服从、哪些地方应该主动暴露未知项。

太具体 LOCK-IN

Agent 可能忠实完成指令，即使实施中已经出现证据，说明应该换一条路。

风险：把地图误当成领地，优化一个已经不成立的计划。

太模糊 DEFAULTS

Agent 会用行业最佳实践补全空白，但一般意义上的好做法未必适合这个代码库和任务。

风险：大量看似合理的小假设，最终组成一个不属于你的方案。

更有效的上下文：说明你处于什么思考阶段、对问题和代码库熟悉到什么程度，以及遇到未知项时应该停下提问、保守前进，还是先做原型。

04 · ARTIFACT LOOP

用一串便宜 artifacts，把未知项赶在昂贵返工之前

这些方法不是每次全部使用的模板，而是一套工具箱。选择标准很简单：现在处于哪个阶段，最可能遇到哪类未知项，哪个 artifact 能最快让它变得可见？

BEFORE IMPLEMENTATION

实施前：先修正认知，再投入代码

这一阶段最便宜的产物不是代码，而是更好的问题、可比较的方向、可指向的参考，以及把关键决策放在前面的计划。

未知的未知 Blind spot pass 让 Agent 主动找出你不知道该问的问题、质量标准和历史坑位。

适合陌生模块、陌生领域和「不知道好是什么样」的任务。关键是同时告诉 Agent 你是谁、目前懂到哪里。

「我要在这个代码库里新增 auth provider，但完全不了解 auth 模块。请做一次 blind spot pass，找出相关的 unknown unknowns，并告诉我怎样才能更好地给你上下文。」

未知的已知 Brainstorms & prototypes 先做差异足够大的方向或低成本原型，通过反应提炼隐性标准。

原型的目的不是提前写正式代码，而是让一个小小的 spec 差异在成本还低时暴露。设计类任务尤其适合先给出多个方向。

「先不要接任何真实数据。用单个 HTML 文件做 4 个差异很大的 dashboard 方向，我会告诉你哪些地方有感觉，以及为什么。」

已知的未知 Interviews 让 Agent 一次只问一个真正会改变架构或范围的问题。

在 brainstorm 之后，你通常已经知道还有哪些模糊处，但还没做决定。限制「一次一个问题」能避免问题清单变成新的认知负担。

「请一次问我一个问题来澄清歧义，优先问答案会改变架构的部分。」

压缩意图 References 当语言太慢或太贫乏时，直接指出现有实现，源码通常是最丰富的参考。

截图能表达外观，源码还能表达 markup、结构、状态和行为。即使参考实现使用另一种语言，也可以要求复刻其语义。

「vendor/rate-limiter 里的 Rust crate 正好实现了我要的 backoff 行为。读它，然后在 TypeScript API client 里复刻同样的语义。」

决策表面 Implementation plans 把最可能被你修改的决策放到计划前面，机械性工作放在后面。

计划不是为了证明 Agent 能预见全部细节，而是为了让数据模型、类型接口、用户流程等高杠杆决策尽早接受审阅。

「用 HTML 写一份实施计划，先列我最可能想改的决策：数据模型、类型接口和用户可见流程。机械性重构放到最后。」

DURING IMPLEMENTATION

实施中：允许即兴，但把偏航留下来

再好的计划也会遇到边界情况。目标不是让 Agent 每次都停住，而是约定一个安全默认，并把脱离原计划的决策记录下来。

持续校准 Implementation notes 遇到迫使方案改变的边界情况时，保守前进，同时记录偏离、原因和选择。

临时的 implementation-notes.md 让你在下一轮实施时不必重新发现同一批未知项，也让 review 不只剩一个失去上下文的 diff。

「维护 implementation-notes.md。若边界情况迫使你偏离计划，选择保守方案继续，在 Deviations 下记录发现、决策和原因。」

AFTER IMPLEMENTATION

实施后：把新地图交给 reviewer，也交还给自己

完成代码不等于完成理解。解释 artifact 负责迁移上下文，quiz 负责验证你是否真的拥有了这张新地图。

共享未知项 Pitches & explainers 把 prototype、spec 和实施笔记合并成 reviewer 能快速获得 buy-in 的单一文档。

好的解释文档既帮助不了解背景的人快速进入，也向专家证明常见失败点已经被考虑。作者建议 demo 优先。

「把 prototype、spec 和 implementation notes 打包成一个能发到 Slack 的文档，开头先放 demo GIF，并提前回答 reviewer 最可能提出的异议。」

验证理解 Quizzes 基于真实变更出题，错题指回对应解释，确认你理解后再 merge。

diff 只能给出浅层线索，行为往往依赖旧代码路径。测验把「我看过了」升级成「我能解释发生了什么」。

「先生成一份包含背景、直觉和实际变更的 HTML 报告，再在底部测试我。只有我全部答对才算可以 merge。」

这次发现的 unknowns，会成为下一次更准确的地图。

05 · FABLE LAUNCH

一个真实案例：什么时候该让 Agent 做，什么时候该让 Agent 教

作者用 Claude Code 编辑 Fable 发布视频。这个过程不是一路「再生成一版」，而是根据未知项类型不断切换协作方式。

01 · LEARN THE LIMIT

转录剪辑够不够准？

作者知道自己不确定 Whisper 与 ffmpeg 能否精确去掉语气词和长停顿。先让 Claude 解释技术边界，这是一个已知的未知。

02 · PROTOTYPE

字幕 UI 能否逐词同步？

这不是靠讨论能消除的不确定性。作者让 Claude 用 Remotion 与转录做原型，把可行性变成可以观看和判断的证据。

03 · EXPAND TASTE

为什么画面显得灰？

最初想做几版调色来挑，但很快发现自己根本不知道「好」是什么。于是停止盲选，改为让 Claude 教调色，处理未知的未知。

关键转向：当你缺少判断标准时，更多变体只会制造更多选择。先让 Agent 扩展你的认知与品味，再回来做方案，反馈才有方向。

06 · NEXT PROJECT

下一次，不妨先让 Agent 帮你找到不知道的事

这是文章可以直接带走的起手式。它不要求你预先知道所有问题，只要求你诚实说明自己的起点，并约定未知项如何被暴露。

可复制的开场 prompt

把方括号内容换成你的项目。重点不是措辞，而是让 Agent 知道你的知识边界与决策规则。

「我正在做 [任务]。我对 [领域 / 模块] 的熟悉程度是 [起点]，目前确定的是 [已知的已知]，还没决定的是 [已知的未知]。请先做一次 blind spot pass，找出我可能没意识到的风险、质量标准和历史约束。遇到会改变架构的未知项时先问我；可以保守处理的小偏差请记录到 implementation notes 后继续。」

先问自己三件事

  1. 我不知道什么？ 哪些问题已经能被命名？
  2. 我只会在看到后判断什么？ 是否应先做多个方向或原型？
  3. 我连好坏标准都没有的是什么？ 是否应该先学习，而不是继续生成？

每一个 explainer、brainstorm、interview、prototype 和 reference，都是在修复成本变高之前，低成本发现「你原本不知道自己不知道」的办法。
